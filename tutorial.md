# MongoDB Atlas Vector Search Setup — Tutorial

This guide walks you through creating the vector index required for Yaksha FAQ's semantic search. You need to do this **once** in the MongoDB Atlas web UI before running `npm run seed`.

---

## What is a vector index?

A vector index lets MongoDB perform `$vectorSearch` — fast approximate nearest-neighbor search over your stored embeddings. The index maps 768-dimensional vectors (generated by `@xenova/transformers` locally) to disk locations so searches don't need to scan every document.

---

## Step 1 — Go to your Atlas cluster

1. Log in to [cloud.mongodb.com](https://cloud.mongodb.com)
2. Select your **cluster0** project
3. Click **Clusters** in the left sidebar

---

## Step 2 — Navigate to Search

Click the **Search** tab (you may see "Search" or a magnifying glass icon) near the top of the cluster view.

> Note: Vector Search is **free** on M0 (free tier) as of 2024+. No cluster upgrade needed.

---

## Step 3 — Create a new index

1. Click **Create Index**
2. Choose **JSON Editor** mode (not the Visual editor)
3. For the collection, select:
   - Database: `yaksha_faq`
   - Collection: `yaksha_faq_faqs`

---

## Step 4 — Paste the index definition

Paste this into the JSON editor:

```json
{
  "fields": [
    {
      "type": "vector",
      "path": "embedding",
      "numDimensions": 768,
      "similarity": "cosine"
    },
    {
      "type": "filter",
      "path": "status"
    },
    {
      "type": "filter",
      "path": "category"
    }
  ]
}
```

Settings explained:
- `type: "vector"` — this is a vector index (required for `$vectorSearch`)
- `path: "embedding"` — the field in your FAQ documents that stores the 768-dim vector array
- `numDimensions: 768` — must match the embedding model's output dimension (Xenova/multi-qa-mpnet-base-dot-v1 produces 768)
- `similarity: "cosine"` — use cosine similarity (recommended for text embedding search). Alternatives: `euclidean` or `dotProduct`
- `status` + `category` filters — added so `$vectorSearch` can pre-filter by these fields at query time (faster than post-filtering)

---

## Step 5 — Name and configure

- **Index Name:** `vector_index` (lowercase, no spaces — this is what the code looks for)
- **Type:** must be **Vector Search**
- Leave other settings at defaults

---

## Step 6 — Create

Click **Create Index**. Atlas will start building the index in the background — this can take a few minutes for large collections. You'll see a progress indicator.

> While the index is building, you can still start the backend. The index becomes available automatically once it's ready.

---

## Verify the index exists

After creation, you should see `vector_index` listed under the `yaksha_faq.yaksha_faq_faqs` collection in the Search tab. If you don't see it immediately, wait a couple minutes and refresh.

---

## Optional: Add index for community posts

If you plan to use community post search in production (the `communitySearchController` also uses vector search), create a second index:

- **Name:** `community_vector_index`
- **Database:** `yaksha_faq`
- **Collection:** `yaksha_faq_communityposts`

```json
{
  "fields": [
    {
      "type": "vector",
      "path": "embedding",
      "numDimensions": 768,
      "similarity": "cosine"
    },
    {
      "type": "filter",
      "path": "status"
    }
  ]
}
```

---

## Common errors

| Error | Cause | Fix |
|---|---|---|
| `numDimensions mismatch` | Field has wrong dimension | Check `numDimensions: 768` matches actual vectors |
| `field not found` | Collection/field name wrong | Field must be `embedding` in `yaksha_faq_faqs` |
| `index already exists` | Tried to create twice | Delete the old one first, or use it as-is |

---

## After the index is ready

```bash
cd backend

# 1. Create other (non-vector) indexes
npm run migrate

# 2. Seed FAQs and generate embeddings (first run downloads ~500MB model)
npm run seed

# 3. Start dev server
npm run dev
```

The seed script will generate embeddings for all 130 FAQs using `@xenova/transformers` and insert them into MongoDB. The vector index will then index them for fast similarity search.
